Multiprocessing
---------------
	
	By default, node.js is single threaded.

	One thread -> Event loop -> handles all requests

	Problem:
		- CPI Heavy task blocks whole eventloop
		Ex:
			- Image processing
			- Large calculations

	Solution:
		Multiprocessing

		The solution is to run multiple processes in parallel.

		Main process -> child process -> Parallel execution

Built in module - 'child_process'

	spawn - Used to run large processes or streaming data

			- Returns stream (stdout, stderr)
			- Efficient for large data

	fork
		A special mehtod for Nodejs processes

		- Used for node scripts only
		- Enable IPC (inter process communication)




	exec
		Runs commands in the shell

		- Buffers entire output
		- Simpler
		- Not good for large data.

		Usecase:
			- small commands executions as separate process
			- Quick scripts


	execFile		
		Runs an executable without a shell

		- Faster than exec
		- More secured

		Usecase
			- Running binary files
			- Secure execution


Node Cluster?
	Cluster allows Nodejs to:
		Use multiple CPU Cores by creating multiple processing (workers)

		Cluster = Run multiple Nodejs processes to handle more load

		1 cpu core -> 1 node process -> Limited performance

		11 core CPU -> Node uses only 1 core

		Remaining 7 core are wasted.


		Solution?
			1 master process

				Multiple worker processes (per CPU core)


				Each worker:
					- Runs your app
					- Handles requests

		Master Process:
			- Manage workers
			- Distributes requests

		Worker processes:
			- Runs your server
			- handle requets

Index in MongoDB

Index - special data structure that improves the Speed of data retrieval operation

Index stores value of a field in sorted strcture (B- Tree) along with references to the documentation that contains those value.

Without Indexes, MongoDB performs Collection Scan (COLLSCAN)
With Indexes, MongoDB performs Index Scan (IXSCAN)


Why INdexes are important:
 Improves performance for:
 	- searching documents
 	- Sorting results
 	- Aggregation operations
 	- Join operations ($lookup)

 However, indexes also have trade-offs:
 	- Extra storage
 	- Slightly slower inserts and updates


By default, mongodb creates index for _id field


Types of Index:
	1. Single Field Index

		db.students.createIndex({city:1})


	2. Compound index
		Index on multiple fields.

		db.students.createIndex({course:1, age:-1})

	db.students.find({couse:'Computer science'}).sort({age:-1})

	Rule:
		Index order matters:
		{course:1, age:1}

		works for seach: 
			course
			course + age

		Not for
			age alone

	3. Unique Index
		Ensures field values remain unique

		db.students.createIndex(
			{email: 1},
			{unique: true}
	 	)

	 	Duplicate insert will fail

	 4. Text Index 
	 	Used for Full Text search

	 	db.articles.createIndex({content: 'text'})

	 	db.articles.find({
	 		$text: {$search: "mongodb"}
	 	}) 

	 5. TTL Index (Time-To_Live)
	 	Automatically deletes documents after specific time

	 	db.sessions.createIndex(
	 		{createdAt: 1},
	 		{expiredAfterSecionds: 3600}
	 	)


	 	Document expire after 1 hour, will be deleted from db

Triggers:
---------
	Trigger is a database object that automatically executed before or after a sepecific event occurs on a table.

	Events:
		- INSERT
		- UPDATE
		- DELETE

	Triggers - automatically , not manually.

 Key Characteristics:
 	- Automatically executed
 	- Associated with single table
 	- Fired on INSERT/UPDATE/DELETE
 	- Can BEFORE or AFTER an event
 	- Uses NEW and OLD Keywords
 		UPDATE - NEW  - valuse after change (OLD- value before change)
 		INSERT - OLD - Not allowed New available
 		DELETE - OLD available and NEW not available

 	- cannot accept parameters or return values
 	- Executes as part of transaction

Syntax:
	
	CREATE TRIGGER trigger_name
		BEFORE | AFTER INSERT | UPDATE | DELETE

	ON table_name
		FOR EACH ROW  --> Triggers runs for every affected row
	BEGIN
		SQL Statements;
	END

	DELIMITER;


SIGNAL SQLSTATE - Manually raise an error or warning condition in storedprocedure/trigger . 
45000 --> generic 'unhandled user-defined excetion'

Practical applictions of Triggers:
	1. Logging
	2. Data validation
	3. Audit trails
		keep track of who changed what and when
    4. Automatic data synchronization


Two types of triggers:
	1. Row - Level Trigger
		Executes once for each row affected by the DML statement.
		If statement affect N rows, trigger fire N times.

	2. Statement - level trigger
		Executes once per SQL statement, regardless of how many rows are affected

		once per statement
		DO not use FOR EACH ROW
		CAN NOT ACCES TO OLD ADN NEW
		Used for summay logging or bulk operations

	trigger should not modify same table it is attached to

	It defines:
		- Wheter MySQL merges the view query?
		- Or creates a temporary table?
		- Or decies automatically?

		- MySQL decies whether to use MERGE or TEMPTABLE

	ERD - Entity Relationship Diagram 

		Visual representation of a database design that shows:
			- What data is stored (entities)
			- How data is described (attributes)
			- How data is connected (relationships)

		ERD Acts as blueprint of database before it is implemented.

	Why ?
		- Helps to udnerstand business requirements
		- Improves communication between technical and non-technical stakeholders
		- Reduce design errors
		- Serves as documentation for database structure
		- Helps convert business logic into database schema.


Key components of ERD
	1. Entity
		Real world object or concept that has independent existence and about which data is stored in database

		Each entity ususally becomes table in database


		Types of entity:
			a. Strong entity

				Entity that 
					- exist independently
					- Has its own primary key
					- Doesn't depend on any other entity for its existance


			b. weak entity

				Entity that
					- Cannot exist independently
					- Depends on a strong entity for its existance
					- Doesn't have a complete primary key on its own

				Example:
					- OrderItem (depends on Order)
					- Invoice LI (depends on Invoice)

			c. associative entity
				Entity created to:
					- Resolve relationships as a table
					- contains foreign keys from participating entities
					- may contains additional attributes

				Enrollment
				Student - Course


				Student 		course			student_course(join table)
												student_id course_id

				Order_Product



	2. Attribute
		An Attribute is a property or characteristics that describes an entity

		Examples:
			- StudentName
			- EmployeeSalary
			- ProductPrice


		Attribute becomes columns in table

		1. Simple Attribute
			- Atomic
			- Indivisible
			- Can not be brokendown further

		Ex:
			Age, Salary, Gender

		2. Composite Attribute
			Can be broken down into smaller sub-attributes

			Name --> First Name, Last Name
			Address --> Street, City, State, ZIP

		3. Derived Attribute
			- Calculated from another attributes
			- Usually not stored physically

		Ex: Age (derived from DOB)
			- Total Amount (Derived from qty * price)

		4. Multivalued attribute
			- Hold multiple values for single entity

			Ex. Phonenumbers (1,2,3)
			- email_add(..,..,..)
			- skills(..,..,..,)

			Multiple attribute handling as separate table

		5. Single valued atttribe
			- Exactly one value for an entity

			Ex: 
				DOB
				EmpId
				SSN


	3. Relationship

		Cardinality
			- Number of instances of one entity associated with another

			1. One-One-One (1:1)

				One entity is related exlactly one instance of another entity

				Person - Passport
				User - Profile


			- Often merged into single table 

			2. One-To-Many (1:M)
				One entity is related to multiple instances of another entity

				customer - Orders

				Department - Employees

				This is most common used

			- Often managed as separate columns as FK constraint

			3. Many-to-Many(M:M)
				Multiple instances of one entity is associated with multiple instances of another entity

				Student <-> Course
				Product <-> Order

			- Often manged as separate join table


ER Models
	
	ER models are created different level of abstraction during database design

	Each level represent specific purpose and audiance:

	1. Conceptual ER Model
		High level business view of data

		Focus
			- what data is needed, and how it stored



	2. Logical ER Model

			Detailed, structured view of data, while independent of any specific DBMS

			converts business req -> structured data model
			Defines Attributes, keys, relationships

			Still DBMS Independent

	3. Physical ER Model	

			Actual implementaiton of db - specific DBMS

		It transplate model attribute, entity --> table, columns and constraints in db
